# BACKLOG — отложенные технические задачи

## ENG-PROD-HTTPS — публичный доступ engbotai.ru (редиректы / HTTPS / Jino)

### Договорённость

До закрытия этой задачи **основная разработка и проверка игры ведутся локально** (см. `DEVGUIDE.md`). Публикация и стабильный заход с интернета на прод-домен — **после** восстановления корректной схемы доступа.

---

### Симптомы

- В браузере: **ERR_TOO_MANY_REDIRECTS**, в Network — чередование **301** и **302**, тип `document / Redirect`.
- С VPS по SSH: `curl -sI --max-redirs 10 https://engbotai.ru` часто даёт **один ответ `HTTP/1.1 200 OK`**, `Server: nginx/1.25.5` — путь запроса с сервера **не обязан совпадать** с путём браузера из интернета (hairpin NAT, другой вход в nginx).

---

### Инфраструктура (зафиксировано на момент диагностики)

| Элемент | Сведения |
|--------|----------|
| Домен | `engbotai.ru`, поддомены (`www`, `pgadmin`, `portainer`, `netdata`) |
| Хостинг | Jino VPS, в панели привязка домена к VPS (Ubuntu) |
| Выделенный IP | `81.177.141.253` |
| Jino «Проксирование» | **Внешние порты 80 и 443 → внутренний порт 80** (TLS завершается на стороне Jino, до VM приходит HTTP на `:80`) |
| DNS | NS делегированы на Cloudflare (`edna.ns.cloudflare.com`, `everton.ns.cloudflare.com`), A-записи на IP VPS |
| Резолв с VPS | `getent hosts engbotai.ru` → `81.177.141.253` (прямой ориджин, без IP Cloudflare в этом ответе) |
| Cloudflare | В панели режим шифрования **Full**; при диагностике имел смысл проверить **прокси (оранжевое облако)** и **Redirect / Page Rules** на предмет петли **www ↔ apex** |
| Docker на сервере | Каталог приложения: `/opt/tennis1v1`, compose: `docker-compose.prod.yml`, nginx монтирует `./nginx/nginx.conf` |

---

### Причины и правки в репозитории (хронология)

1. **Сертификат `*.jino.ru` при обращении по имени хоста** — трафик на 443 не доходил до своего nginx; частично связано с отсутствием привязки домена к VPS в панели Jino (позже домен привязали).

2. **Петля редиректов при схеме Jino 443→80** — nginx на порту 80 делал **безусловный** `return 301 https://…`. Клиент снова открывал HTTPS → Jino снова слал HTTP на 80 → цикл.

3. **Попытка учитывать `X-Forwarded-Proto: https`** — на практике заголовок **не приходил** от Jino, условие не срабатывало, редирект оставался.

4. **Итоговое направление в коде (ветка `main`)** — на **порту 80** для основных виртуальных хостов **убраны** редиректы на HTTPS; отдаётся тот же контент, что и с TLS на стороне шлюза. Для бэкенда на 80 используется логика `X-Forwarded-Proto` с подстановкой **`https`**, если заголовок пустой (`map` → `$client_scheme` в `nginx/nginx.conf`). Редирект **www → голый домен** сохранён.

5. **Критичный инцидент на проде** — контейнер nginx отдавал **старую** конфигурацию с **`$redirect_http_to_https`** после того, как в GitHub уже лежала новая версия. Причина: на сервере **не обновляли** `origin` (`git pull` без предварительного `git fetch` / устаревшее «up to date»). Пока в `nginx -T` встречается `redirect_http_to_https`, на диске сервера **не** актуальный `nginx/nginx.conf`.

### Файлы и команды для возобновления работы

| Назначение | Где / что |
|------------|-----------|
| Конфиг reverse proxy | `nginx/nginx.conf` |
| Прод compose | `docker-compose.prod.yml`, сервис `nginx` |
| Деплой | `scripts/deploy.sh` |
| Проверка, что в контейнере нет старой логики | `docker compose -f docker-compose.prod.yml exec nginx nginx -T 2>/dev/null \| grep -c redirect_http_to_https` → ожидается **0** |
| Проверка файла на хосте | `grep redirect_http_to_https /opt/tennis1v1/nginx/nginx.conf` → пусто |
| Обновление кода на сервере | `cd /opt/tennis1v1 && git fetch origin && git pull origin main` |
| Перечитать nginx | `docker compose -f docker-compose.prod.yml exec nginx nginx -s reload` |
| Проверка с **ПК** (не с VPS) | `curl -vI --max-redirs 20 https://engbotai.ru` — смотреть цепочку `Location:` |

### Гипотезы на доразбор (когда вернёмся к задаче)

- Петля **не в nginx**, а в **Cloudflare** (два правила редиректа: apex↔www) или в **поведении Jino** вне конфигурации контейнера.
- Различие **IPv6 (AAAA)** vs IPv4 у браузера.
- Кэш / HSTS в браузере после длительных экспериментов — проверка с другого устройства или LTE.

### Возможное улучшение инфраструктуры

- В панели Jino: если доступно, настроить **внешний 443 → внутренний 443** до Docker-nginx с Let’s Encrypt, а **внешний 80 → внутренний 80** для ACME и HTTP.

---

### Критерий готовности (Definition of Done)

- С рабочего ПК и с мобильного интернета `https://engbotai.ru` открывается без ошибки редиректов, игра и API доступны согласно архитектуре проекта.
- `curl -sI --max-redirs 10 https://engbotai.ru` с ПК — один успешный ответ без цикла.
- В прод-контейнере nginx нет устаревшей схемы с `redirect_http_to_https`.

## ENG-GAME-AUDIO-FILES — звуки матча из файлов вместо синтеза

### Контекст

Сейчас звуки во время матча воспроизводятся через **Web Audio (осцилляторы)** в `client/src/game/matchAudio.ts`: это закрывает бизнес-требование «на каждое событие есть звук», но **не** соответствует целевой схеме из `ARCHITECTURE.md` / GDD (готовые файлы в `client/public/assets/sounds/`, пакеты Kenney и т.п.).

### Задачи

- Подобрать или подготовить набор **коротких** несжатых/OGG файлов под перечень из эпика 03 (удар, отскок, сетка, аут, эйс, гейм, сет, матч, смена сторон и др. по необходимости).
- Подключить загрузку и воспроизведение (Phaser cache / `Audio` / единый маленький плеер), сохранить текущую матрицу «какое событие — какой звук».
- Учесть лицензии и кредиты в `ARCHITECTURE` / README при необходимости.
- Опционально: громкость и отключение звука в настройках игрока (если появятся в продукте).

### Критерий готовности

- В матче не используется синтетический `OscillatorNode` для перечисленных игровых событий; звуки идут из файлов в репозитории (или из явно описанного CDN), сборка клиента их включает.

## ENG-GAME-BOUNCE-EVENT — точный отскок мяча (опционально)

### Контекст

Отскок сопровождается звуком на клиенте по **эвристике** (смена знака `vy` между кадрами `game:state`). При редких режимах симуляции или лагах возможны ложные срабатывания или пропуски.

### Задачи

- При необходимости: расширить протокол (например, разовое событие `game:sound` или флаг в `game:state`) с сервера в момент касания корта; клиент только воспроизводит.
- Согласовать с `docs/dev/API.md` и не ломать совместимость без версии протокола.

### Критерий готовности

- Звук отскока стабильно совпадает с касанием корта в авторитетной симуляции; эвристика по `vy` удалена или оставлена только как fallback.

## ENG-MATCH-HISTORY-DB-RETENTION — жёсткий потолок матчей в БД

### Контекст

История в API ограничена `take: 100`; старые строки `Match` / `MatchPlayer` в PostgreSQL не удаляются.

### Задачи

- Определить политику (например, после вставки матча удалять записи старше N-го по участнику или глобальную уборку).
- Реализовать в транзакции с `persistOnlineMatch` / `persistBotMatch` или отдельным воркером.
- Документировать в `docs/dev/ARCHITECTURE.md` при необходимости.

### Критерий готовности

- Размер таблиц матчей не растёт бесконечно при длительной эксплуатации (или явный лимит задокументирован как осознанный отказ).
